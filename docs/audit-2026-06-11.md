## Systemic root cause (read first)

Three design decisions combine into most of the security findings:

1. **Every Next.js route handler talks to Strapi with the full-access admin token.** In `api.ts:74-145`, `find/findOne/create/update` fall back to `STRAPI_API_TOKEN` whenever no user JWT is passed — and **no handler ever passes the user's JWT**. So Strapi enforces *zero* authorization.
2. **The backend has no ownership policies.** `test-attempt`, `user-answer`, `writing-submission`, `speaking-submission`, `test-progress`, `full-mock-test-attempt` all use default core routers/controllers with no `is-owner` policy.
3. **The only thing protecting user data is a manual `filter: { user: { id } }` line in each handler** — which several handlers forget.

**The fix that collapses the most findings:** define Strapi ownership policies on user-scoped collections and forward the user JWT, so authorization is enforced server-side by default rather than per-handler by memory.

## 🔴 Critical (7 distinct issues)

- [x] **C1 — The IELTS answer key is publicly downloadable.** *(flagged twice, independently)* — **FIXED & VERIFIED in `backend/src/index.ts`** (revoked Public+Authenticated access to `question` and the content types; populate-leak closed by also revoking passage/section reads). Verified live: `GET /api/questions` → 403, `GET /api/reading-passages?populate=questions` → 403. Applies on next backend/prod restart. NOTE: did *not* mark fields `private` — that breaks server-side scoring, which reads `correct_answer` over admin-token REST.
  `backend/src/index.ts:28-36` grants the **Public** role `find`/`findOne` on `api::question`, and `question/schema.json` stores `correct_answer` + `explanation` as plain, non-private fields with the default controller. Anyone, unauthenticated, can call `GET /api/questions?fields[0]=correct_answer&pagination[pageSize]=1000` (or reach it via `reading-passage`/`listening-section` populate) and pull the complete answer key for every reading/listening test **before submitting**. The careful `fields` stripping in the `start` route handlers is moot because the client can hit Strapi directly. Defeats the integrity of every mock exam.
  **Fix:** Remove `question` (and question-populating passage/section endpoints) from public perms; mark `correct_answer`/`explanation` as `config.private: true`; serve all test content only through the Next route handlers.

- [x] **C2 — IDOR: any logged-in user can read/modify every other user's attempts, answers, and graded submissions.** — **FIXED & VERIFIED in `backend/src/index.ts`**. Revoke is by content-type UID *prefix* (deletes every action incl. `delete`, format-agnostic) because Strapi's real actions are `api::test-attempt.test-attempt.find`, not `api::test-attempt.find`. **Discovery during verification:** the live grants were set via the **admin UI** (not the old bootstrap) and included `delete` on `writing-task` and `full-mock-test-attempt` — i.e. any authenticated user could *delete* writing prompts and others' full-mock attempts. All now revoked. Verified live (`test-attempts`/`writing-submissions`/`writing-tasks`/`full-mock-test-attempts` → 403) and against the `up_permissions` table (only `public business-inquiry.create` remains). Applies on next backend/prod restart. Follow-up hardening still worth doing: `is-owner` policies + forward user JWT as defense-in-depth.
  `test-attempt/routes/test-attempt.ts` and siblings use default routers with no ownership filter, and the Authenticated role has `find`/`findOne`/`update`. `GET /api/test-attempts`, `/api/user-answers`, `/api/writing-submissions`, `/api/speaking-submissions` return **all users'** band scores, essay text, and speaking transcripts (up to 1000/page). `.update` grants let them tamper with others' scores.
  **Fix:** Add an `is-owner` policy injecting `filters.user = ctx.state.user.id` on every user-scoped collection; remove blanket `find`; set `test-attempt.user` to `required: true` stamped server-side.

- [x] **C3 — Telegram login-code table is readable by any authenticated user → account takeover + PII leak.** *(flagged twice)* — **FIXED & VERIFIED in `backend/src/index.ts`** (revoked all Authenticated access to `telegram-auth-code`; only the internal verify controller touches it). Verified live: `GET /api/telegram-auth-codes` (authed) → 403. Applies on next backend/prod restart. Follow-up (separate findings): hash codes at rest + bind code to session (C5).
  `telegram-auth-code/schema.json` stores `code`, `telegram_id`, `phone` in plaintext, and the Authenticated role has `find`/`create`/`update`. Since Strapi self-registration is open, an attacker registers, gets a JWT, then `GET /api/telegram-auth-codes?filters[used][$eq]=false` to dump every live 60-second login code with its owner — then redeems it. Also harvests all users' phone numbers.
  **Fix:** Remove all REST CRUD grants on `telegram-auth-code` (only consumed internally by the verify controller); mark fields private; hash codes at rest.

- [x] **C4 — Telegram OAuth callback authenticates users from an *unverified* JWT.** — **FIXED by deletion.** Confirmed the OIDC flow was disabled legacy (real login is the 6-digit code flow): `signInWithTelegram` was already commented out and nothing rendered the login widget. Deleted the dead `api/auth/telegram/{callback,widget,mini-app}` routes + the unused `telegram-login-widget` component, and removed the commented `signInWithTelegram` block. This also eliminated two mediums that lived only in the callback: the deterministic-password derivation and the token/id_token logging (see below). No impact on live logins (6-digit, Google, email/password).
  Note: the practical exploitability was lower than first stated — reaching the unverified decode still required a real `oauth.telegram.org` token exchange — but deleting removes the footgun entirely.

- [~] **C5 — verify-code doesn't bind a code to the requesting session.** — **PARTIALLY MITIGATED.** The worst vector (mass-harvest all live codes via the API, per C3) is now closed, so a stolen code requires phishing/relay rather than a bulk dump. Brute-force is also hardened (below). **Residual:** a code is still matched on `code`+`used:false` with no tie to the presenting browser; full binding needs a site-initiated nonce (a small redesign of the login start). Left open intentionally.

- [x] **C6 — SSRF: `speaking/evaluate` fetches an attacker-controlled URL.** — **FIXED & TESTED.** New `lib/safe-audio-url.ts` `resolveSafeAudioUrl()` parses the reference with `new URL` and only allows the Strapi origin with a `/uploads/` pathname (path-traversal rejected post-normalization), or an explicitly allowlisted CDN origin (`MEDIA_ALLOWED_ORIGINS`, empty by default — no config needed for current local-uploads prod). Wired into **both** `speaking/submit` (rejects unsafe `audio_url` up-front with 400) and `speaking/evaluate` (re-validates before `fetch`, protecting legacy rows). Verified with a 12-case runtime test: `169.254.169.254`, `evil.com`, `//evil.com`, `/uploads/../secret`, `api.bandup.uz.evil.com`, and `file://` all rejected; `/uploads/...` and allowlisted CDN accepted. NOTE: did not switch to opaque file ids (larger refactor) — origin+path allowlisting fully closes the SSRF.

- [x] **C7 — Results page leaks any user's results to *unauthenticated* visitors.** — **FIXED** in `(dashboard)/dashboard/results/[attemptId]/page.tsx`: replaced the `attempt && user && …` guard (which evaporated when `user` was null) with an owner/admin check that holds for anonymous visitors, while still allowing unowned guest attempts (preserves the `?claim=` flow). **Also fixed a sibling leak the audit didn't flag:** `(fullscreen)/dashboard/speaking/result/[attemptId]/page.tsx` had **no ownership check at all** — any visitor with an attemptId could read anyone's speaking transcripts/scores; added the same guard. The two full-mock results pages were already safe (`if (!user) notFound()` + owner check). **`middleware.ts` deliberately NOT added**: a blanket dashboard auth-gate would break guest test-taking + guest result viewing; the page-level ownership guards close the actual leak. Verify with a browser run before considering fully done.
  `results/[attemptId]/page.tsx:42-108`: the ownership guard is `if (attempt && user && !isAdmin && attempt.user?.id !== user.id)`. When `user` is null (no/expired JWT) the `&& user &&` short-circuits and the guard is **skipped entirely** — the page renders the attempt's scores, the user's answers, and the correct answers. Attempt documentIds are handed to clients in plaintext (submit responses, `?claim=` URLs), so they're enumerable. There's no `middleware.ts` and the dashboard layout doesn't redirect anonymous users.
  **Fix:** Treat null user as unauthorized for any owned attempt; add `middleware.ts` to gate the dashboard.

## 🟠 High (23 — grouped)

### Authorization / IDOR
- [x] **Full-mock session PATCH has no ownership check + trusts client scores** — `full-mock-test/session/route.ts`. *(Single most-corroborated finding — 3 finders.)* **FIXED.** PATCH now (1) fetches the session and **rejects with 403 unless the caller owns it** (closes the cross-user IDOR), and (2) **ignores client-supplied scores entirely**, deriving each module band from the linked `test_attempts[].band_score` and computing `overall_band_score` from those — so a user can no longer forge their scores. Verified safe against the results page, which already derives module scores from the attempts (it only reads `overall_band_score`, with an attempts-based fallback). Frontend typechecks clean.
- [x] **`attempts/claim` lets any user claim any orphaned guest attempt** — **FIXED & TESTED.** New `lib/guest-claim.ts`: when a guest submits a test (reading/listening/writing), the ownerless attempt's documentId is recorded in a signed (HMAC-`STRAPI_API_TOKEN`), httpOnly cookie. Claiming now requires the attemptId to be present in that verified cookie — enforced in **both** `attempts/claim` (403 otherwise) and the results-page inline `?claim=` path. An attacker can't forge it without the server secret, which also moots the documentId-enumeration vector (no rate limit needed). Verified with an 8-case test incl. forged-signature and tampered-payload rejection. No schema change required.
- [x] **`fullMockAttemptId` linked on submit without ownership check** — `reading/submit`, `listening`, `writing`, `speaking`. **FIXED.** New `lib/full-mock.ts` `ownsFullMockSession()`; all four submit routes now return 403 if a `fullMockAttemptId` is supplied that the caller doesn't own, so a user can't inject their module attempt into a stranger's full-mock session. Typechecks clean.
- [x] **`speaking/submit` trusts arbitrary `testId`/`topicId`/`audioUrl`** — `speaking/submit/route.ts`. **FIXED.** audioUrl SSRF closed (C6); now every topic is fetched and verified to exist, and the test is **derived from the topics** (the client `testId` is ignored), rejecting payloads whose topics don't all belong to one real test (404/400). Frontend typechecks clean.

### AI cost / abuse / injection
- [x] **`videos/quiz` POST is unauthenticated but triggers billable Gemini calls + admin writes** — `videos/quiz/route.ts`. **FIXED** — POST now returns 401 when unauthenticated (it called `getAuthUser` but never checked the result). (GET still serves cached quiz questions anonymously — read-only, no Gemini/writes; low-value quiz answers, left as-is.)
- [x] **`writing/free-write` has no max essay length** — `free-write/route.ts`. **FIXED** — rejects `content` over 20k chars (~3000+ words) with 400 before any Gemini call. NOTE: a global per-user **AI quota / rate limit** is still unimplemented (separate medium: `ai-usage-log` is write-only) — this cap blocks the per-request multi-MB abuse, not sustained call volume.
- [x] **Prompt injection into the scoring prompt** — `evaluate-essay.ts`. **HARDENED** — criterion scores are now clamped to the valid IELTS band range [0, 9] (`clampBand`) before the overall is recomputed, so an injected out-of-range score (e.g. 99) can't inflate the result. (Already had code-side overall recompute + length/off-topic caps; the candidate text is still concatenated into the prompt, but it can no longer push scores out of range.)
- [x] **Sample-answer leak at test start** — `speaking/start-test/route.ts`. **FIXED** — `sample_answers` is no longer fetched or sent to the client at test start; the runner's "reveal sample" block is gated on its presence so it simply doesn't render. (Note: the separate item below about `speaking-topic` being publicly readable in Strapi was already closed by C1's permission revoke.)

### Telegram auth
- [x] **verify-code brute-force protection is weak** — `telegram-auth.ts`. **FIXED & TESTED.** Limiter now counts only *failed* attempts and blocks at 8/15min per IP (with memory pruning); combined with crypto-random, single-use, 60s codes this makes guessing impractical. Verified live: 8×400 then 429 on the 9th. **Residual (noted in code):** still in-memory/per-instance — a multi-instance deployment should move this to shared storage (Redis).

### Correctness (scoring — these silently mis-grade real users)
- [x] **Answer normalizer comma-splits free-text answers** — `reading/submit/route.ts` (and listening). **FIXED & TESTED.** Extracted a shared `lib/scoring.ts` `isAnswerCorrect(questionType, user, correct)`: only `mcq_multiple` is split+sorted as a letter-set; all other types use text normalization, and thousands-separator commas between digits are stripped so `"60,000" === "60000"`. Both submit routes now fetch `question_type` and call it. Verified with a 12-case runtime test against the compiled helper (incl. `60000`/`60,000` both directions, `E,C`==`C,E`, case-insensitivity). NOTE: still does not handle `20%`↔`20 percent` or `colour`↔`color` (separate normalization finding below).
- [x] **Reading `mcq_multiple` scores 0/2 for the intuitive interaction** — `reading-questions.tsx`. **FIXED.** Replaced the per-question `<MultipleAnswer>` widgets with the shared `<MultipleAnswerGroup>` (same component listening uses): one multi-select for the whole group that writes the same combined value (`"C,E"`) to every sub-question, so each row matches its stored `correct_answer` and the natural "pick TWO" scores full marks. Frontend typechecks clean. (Partial credit still not awarded — matches prior behavior; out of scope.)
- [x] **Bonus (Low): blank `correct_answer` no longer scored as correct** — fell out of the `lib/scoring.ts` rewrite; an empty/whitespace key now never auto-passes an empty answer. (Verified in the test suite.)
- [x] **Number-word / British-American / symbol normalization** — `lib/scoring.ts`. **FIXED & TESTED** — `%`/`per cent`/`percent` canonicalized, and a curated British→American spelling map (safe canonicalization, never merges different answers). 18-case test incl. false-positive guards (four≠for, hour≠our).
- [x] **`find()` swallows Strapi errors and returns `[]`** — `api.ts`. **FIXED** — `find()` now throws on a non-2xx response instead of returning `[]`, so a 500/timeout no longer looks like "no data." In the submit routes this means the totalQuestions (passages/sections) fetch failing now aborts before any attempt is created, rather than recording `totalQuestions=0`. (Legitimately-empty results are still `200` + `[]` and unaffected.)
- [x] **Non-atomic submission writes** — `reading/submit`, `listening/submit`. **FIXED** — the answer-insert loop is wrapped in try/catch; on any failure the attempt is rolled back via a new `del()` helper, so we never leave a completed attempt with partial answers, and a client retry produces one clean attempt instead of a duplicate.

### Deployment
- [~] **SQLite + local-disk uploads → data loss** — **PARTLY ADDRESSED + clarified.** Reality check: the *active* deploy is Droplet + pm2 (`.github/workflows/deploy.yml` + `ecosystem.config.js`) on a **persistent disk**, and prod already runs **PostgreSQL** — so DB and uploads do persist there; the "wiped on redeploy" scenario was the broken, frontend-only `.do/app.yaml` (App Platform) path. **Done:** wired `backend/config/plugins.ts` so uploads can go to S3-compatible object storage (DO Spaces) when `UPLOAD_PROVIDER=s3` + the provider package are present, safely falling back to local otherwise. **Remaining (ops, needs your call):** (1) to actually move audio off the disk: `pnpm add @strapi/provider-upload-aws-s3` in backend, set the `SPACES_*` env, deploy; (2) decide the fate of `.do/app.yaml` — if App Platform no longer serves the frontend, delete it to remove the conflicting/ephemeral deploy path (left in place since I can't confirm which host serves bandup.uz).

## Recommended remediation order

1. **Lock down Strapi permissions today** (C1, C2, C3): strip public `question`/passage perms + mark answer fields private; add `is-owner` policies; remove `telegram-auth-code` REST grants. Closes the answer-key leak and the mass-IDOR in one pass.
2. **Fix C7 + add `middleware.ts`** so the dashboard isn't the only auth gate.
3. **Telegram auth** (C4, C5): verify the id_token or delete the OIDC route; bind codes to sessions.
4. **C6 SSRF** + the full-mock PATCH ownership/score-forgery high.
5. **Scoring correctness**: the comma-split bug and reading `mcq_multiple` 0/2 bug are actively mis-grading users right now — high user-trust impact, low fix cost.
6. **AI cost controls**: auth + length caps + a real rate limiter.
7. **Deployment**: Postgres + Spaces upload provider; reconcile the two deploy strategies.
